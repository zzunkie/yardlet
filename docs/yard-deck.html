01문제 — CLI 에이전트 시대의 운영 공백

Codex CLI와 Claude Code는 이미 훌륭한 워커다. 그런데 그 워커를 운영하는 건 전부 사람 몫이다.

세션 = 상태

대화가 곧 메모리라서 터미널을 닫으면 작업 맥락이 증발한다. 긴 작업은 사람이 매번 이어붙여야 한다.

사람이 오케스트레이터

계획 분해, 다음 작업 지시, 결과 검증, 재시도 판단 — 에이전트를 쓰는데 운영 루프는 수동이다.

단일 세션 · 단일 워커

독립적인 작업도 한 줄로 세워 순서대로 돌린다. 워커를 바꿔 끼우는 것도, 동시에 돌리는 것도 수작업.

검증의 부재

"다 됐습니다"라는 자기 보고를 믿는 수밖에 없다. 수용 기준 대비 무엇이 됐는지 기계가 따져주지 않는다.

02Yard란 — 몇 문장이 곧 작업 계약이 된다

평소에 여는 건 Yard다. Codex나 Claude Code를 직접 열지 않는다.

03핵심 원칙

① Zero AI API keys (하드룰)

Yard 코어는 AI 프로바이더 API 키를 요구·요청·저장·호출하지 않는다. 이미 결제 중인 구독 CLI를 부려쓴다. 준비된 워커가 없으면 멈추지, 절대 API로 폴백하지 않는다. 워커 서브프로세스 env에서 빌링 변수를 스크럽하는 가드까지 포함.

② 정책 vs 메커니즘

라우팅 해석은 결정론적·감사 가능한 코드. 텔레메트리는 정책 변경을 제안만 하고 사람이 적용한다. 학습 루프(설계됨)도 같은 규율 — 사이클마다 후보를 모으되, 승격은 사람이.

③ .agents/가 캐노니컬 상태

워커는 콘텐츠를 쓰고, Yard만이 시스템-오브-레코드를 쓴다(단일 상태 레이어). 세션이 죽어도 인텐트·큐·체크포인트·핸드오프는 디스크에 남는다.

④ 레이어드 안전

zero-key(하드) + 패킷의 위험행동 자기게이트(소프트) + evaluator의 금지경로 사후검사 + 워크스페이스 훅(설계됨). 위험한 행동은 needs_user로 멈춰서 사람에게 묻는다.

05플래닝 게이트 — 요청을 계약으로

  • 인텐트 계약: 목표 요약, allowed scope, out-of-scope(엄격), 검증 가능한 수용 기준 트리

06워커 시스템 — 교체 가능, 설정만으로 확장

내장 어댑터

Codex CLI·Claude Code: 검증된 플래그 조립, 샌드박스 매핑(workspace-write ↔ acceptEdits), JSONL 라이브 스트림, 일시 장애 시 세션 재개.

제네릭 어댑터

제3의 구독 CLI는 workers.yaml 편집만으로 추가. args 템플릿 + {run_dir} {model} {effort} {image} 플레이스홀더. 코드 수정 0줄.

운영

홈 화면에서 워커별 on/off 토글. 모델·effort 프리셋은 CLI 자체와 동기화 — codex는 models_cache.json, claude는 --help 파싱. 손으로 관리하는 모델 목록 없음.

결정론 라우팅 + 학습하는 정책

태스크 kind/risk → 후보 워커 → 준비상태 프로브 → 폴백. 모든 런이 텔레메트리로 기록되고 yard routing review가 "claude-code가 리팩터에서 더 잘함" 같은 변경을 제안 — 적용은 언제나 사람(yard routing apply).

zero-key env 스크럽워커별 enabled 플래그per-task model/effortcost bias 다이얼준비상태 프로브 (--version)

07병렬 큐 — worktree 격리 동시 실행

독립 태스크 N개를 각자의 git worktree에서 동시에. 서로 다른 워커로도.

  • 워커는 병렬, 상태는 단일 작성자 — 큐 파일은 오케스트레이터 루프만 쓴다
  • 각 태스크는 yard/<task-id> 브랜치 — 서로의 미커밋 변경을 보지 않는다
  • 통합은 순차, 완료 순서대로 --no-ff 머지
  • 충돌은 자동 해결하지 않는다 → Partial + worktree 보존 + 핸드오프 기록
  • git이 아니거나 더티 트리면 순차로 폴백 — 기본값 max_parallel: 1 (옵트인)
HEAD │ ┌────────┼────────┐ ▼ ▼ ▼ yard/101 yard/103 yard/104 ← worktree ×3, 동시 실행 codex claude codex │ │ │ ▼ ▼ ▼ 평가 → 머지(순차, 완료순) → HEAD' │ 충돌? → Partial + worktree 보존 사용자 진행 중 머지는 절대 건드리지 않음

설정: Settings "병렬 작업 수" · yard run --auto --parallel 3 · 모니터는 런별 탭(Tab/←→)으로 동시 추적.

08실행 사이클과 상태 기계

┌─────────┐ │ queued │◄──────────┐ └────┬────┘ │ 재큐 ▼ │ ┌─────────┐ ┌────┴────┐ │ running │─────▶│ failed │ 자동 재시도 └────┬────┘ └─────────┘ ┌─────────┼──────────────┐ ▼ ▼ ▼ ┌──────┐ ┌─────────┐ ┌────────────┐ │ done │ │ partial │ │ needs_user │ └──────┘ └────┬────┘ └─────┬──────┘ 체크포인트 │ 연속 실행 │ a 로 응답 ┌────┴────┐ │ │자가보고 │→ 자동 │→ 답변이 패킷에 │머지충돌 │→ 사람 │ 실려 재개 └─────────┘
  • Partial = 이어하기, 재실행 아님: 다음 런 패킷에 직전 체크포인트 + 요약 + 미해결 검증 실패 목록이 자동 주입 — "끝난 일은 다시 하지 마라"
  • 자동 드레인은 자가보고 Partial을 자동 연속(시도 캡), 머지 충돌만 사람을 기다림
  • a 키 — 어느 상태든(실행 중·완료 제외) 태스크에 지시를 붙여 실행. NeedsUser면 질문 답변, Partial이면 남은 작업 컨텍스트가 화면에
  • 모든 런이 남기는 것: result.json · handoff.md · checkpoint.md · 평가 · 텔레메트리
  • evaluator: 결과 스키마 검사, id 매칭, drift, 금지 경로(시크릿·워크스페이스 밖) — 자기 보고만으로 done이 되지 않는다

09복원력 — 세션은 죽어도 작업은 죽지 않는다

워커 입양 (adoption)

yard를 꺼도 워커는 살아서 일한다. 재시작하면 살아 있는 고아를 입양 — 태스크는 Running 유지, 모니터가 로그를 이어 테일, 결과가 떨어지는 순간 평가·머지. 중복 워커를 띄우지 않는다.

플래닝 결과 회수

플래닝 중 종료해도 워커가 마저 쓴 planning-result.json을 다음 시작 때 자동 소비. 더 새 플랜이 있으면 낡은 결과가 현재 작업을 덮어쓰지 못하게 폐기(supersession 가드).

worktree 인지 복구

병렬 런이 고아가 돼도 결과가 있으면 worktree까지 머지해서 회수. 미완이면 worktree 정리 후 재큐. 머지 abort는 자기 자신의 머지만 — 사용자의 진행 중 머지는 보호.

제자리 업그레이드

실행 중 바이너리가 교체되면 상태줄이 알리고 u 한 번으로 exec 재시작 — 상태는 전부 디스크라 잃는 것이 없다. yard recover로 수동 복구도.

전 경로(스폰→크래시→복구→평가→통합) 유실 감사 완료 — 발견된 4개 유실/손상 케이스 전부 수정, git 통합 테스트로 고정.
