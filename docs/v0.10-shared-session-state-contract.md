# V010-001 공유 session 및 state 계약

> 상태: normative architecture contract
>
> 적용 범위: V010-001과 이를 소비하는 V010-002, V010-003, V010-004,
> V010-005, V010-006, V010-008
>
> 비적용 범위: 이 문서는 production runtime 변경, schema migration,
> TUI/GUI 설계, CLI flag, provider별 invocation을 구현하지 않는다.

## 1. 목적과 규범 언어

이 문서는 planning session, task channel, worker attempt, event, action,
draft state, question/answer, artifact, completion, runtime resource가 하나의
의미 계층을 공유하도록 하는 V010-001의 source of truth다. CLI, TUI, GUI와
worker adapter는 이 계약을 각각 해석하는 별도 상태 머신을 만들지 않고,
동일한 core action을 호출하고 동일한 canonical record를 projection한다.

`MUST`, `MUST NOT`, `SHOULD`, `SHOULD NOT`, `MAY`는 규범 요구사항이다.
규범 문장과 현재 구현이 다르면 규범 문장이 목표 계약이며, 차이는 이
문서의 명시적 gap이다. 이 문서 자체는 구현 완료를 약속하지 않는다.

## 3. 핵심 불변식

1. **One model, many projections.** 모든 surface는 같은 entity, event,
   action, lifecycle을 MUST 사용한다.
2. **Core is the sole canonical writer.** user와 worker는 intent와 content를
   author할 수 있지만, id, sequence, transition, activation과 canonical
   persistence는 Yardlet core만 MUST 기록한다.
3. **Task continuity, attempt accountability.** 하나의 durable task channel은
   retry, fallback, resume, reviewer attempt 전체를 포괄해야 한다. 각 attempt의
   worker identity, worker session, raw stream, result와 artifact는 서로 섞이면
   안 된다.
4. **Explicit confirmation.** planning draft는 해당 revision을 지목한
   `confirm` action이 commit되기 전까지 어떤 경로로도 runnable queue가 될 수
   없다.
5. **Raw evidence survives normalization.** normalized event는 raw stream의
   위치를 참조해야 하며 raw evidence를 덮어쓰면 안 된다.
6. **Canonical facts precede indexes.** channel/resource/search/latest index는
   삭제 후 canonical facts만으로 재생성할 수 있어야 한다.
7. **Fail closed on missing semantics.** runnability, ownership, confirm,
   destructive resource lifecycle에 필요한 의미가 없거나 모순되면 read-only
   projection만 허용한다.
8. **Replay is deterministic.** wall clock, filesystem enumeration order,
   provider output timing, UI arrival order는 replay ordering 근거가 될 수 없다.

## 4. 공통 vocabulary

### 4.1 Entity 관계

```text
Workspace
  └── Session
      ├── DraftRevision* -> Confirmation -> ActiveIntent -> WorkQueue
      ├── Event* <-> ActionReceipt*
      └── TaskChannel*
          ├── Task
          ├── WorkerAttempt*
          │   ├── RawStream
          │   ├── WorkerSessionRef?
          │   ├── Question* -> Answer?
          │   └── Artifact* / RuntimeResource*
          └── Completion?
```

`*`는 0개 이상이다. `TaskChannel`은 저장된 독립 mutable state machine이
아니라 `(intent_id, task_id)`에 속한 canonical facts의 durable projection이다.
따라서 attempt가 바뀌어도 channel은 하나이고, attempt history는 분리된다.

### 4.2 Entity 정의

| Entity | 규범적 의미 | 최소 canonical linkage | owner |
|---|---|---|---|
| `Session` | planning 또는 confirmed work를 둘러싼 durable interaction scope | `session_id`, `workspace_id`, `kind`, `lifecycle` | core |
| `DraftRevision` | 아직 실행 권한이 없는 intent와 plan의 immutable revision | `draft_revision_id`, `session_id`, `parent_revision_id?`, `content_digest` | user/worker authors, core records |
| `Confirmation` | 정확한 draft revision을 active intent와 queue로 승격시키는 commit receipt | `confirmation_id`, `action_id`, `draft_revision_id`, output digests | user authorizes, core commits |
| `ActiveIntent` | confirmed user contract snapshot | `intent_id`, `confirmation_id` | core |
| `WorkQueue` | confirmed task execution snapshot | `queue_id`, `intent_id`, `confirmation_id` | core |
| `Task` | session보다 오래 실행될 수 있는 durable work unit | `task_id`, `intent_id`, `materialized_by_confirmation_id` | core |
| `TaskChannel` | task의 user-facing continuous history projection | deterministic `channel_id`, `intent_id`, `task_id` | derived by core |
| `WorkerAttempt` | 한 worker invocation 또는 명시적 continuation 단위 | `attempt_id`, `task_id`, `worker_id`, `worker_session_ref?` | core |
| `Event` | 발생한 사실의 immutable ordered envelope | `event_id`, `session_id`, `seq`, `type`, causality | core |
| `ActionReceipt` | surface가 요청한 core operation과 그 결정적 결과 | `action_id`, request digest, status, result ids | core |
| `Question` | user input을 요구한 정확한 channel position | `question_id`, `task_id`, `attempt_id`, `asked_event_id` | worker authors, core records |
| `Answer` | 하나의 open question에 대한 user turn | `answer_id`, `question_id`, `action_id` | user authors, core records |
| `Artifact` | work가 만든 durable file/evidence 결과 | `artifact_id`, `task_id`, `attempt_id`, path/digest | producer authors, core records |
| `RuntimeResource` | terminal, process, service, browser처럼 현재 생명성이 변하는 대상 | `resource_id`, `task_id`, `attempt_id`, owner, observations | producer declares, core records/probes |
| `Completion` | evaluation을 포함한 task의 accountable terminal outcome | `completion_id`, `task_id`, `attempt_id`, outcome, evidence refs | core |

### 4.3 Session 종류와 lifecycle

`Session.kind`는 `planning` 또는 `work`다. 하나의 planning session이 confirm되면
같은 `session_id`를 유지한 채 `kind=work` projection을 추가할 수 있다. 새 id로
갈라지는 구현도 가능하지만 이 경우 `continued_from_session_id`가 MUST 있어야
한다. surface는 id 전환을 channel 단절로 보여서는 안 된다.

| lifecycle | 의미 | 허용된 다음 상태 |
|---|---|---|
| `open` | action을 받을 수 있다. | `awaiting_user`, `confirmed`, `closed`, `failed` |
| `awaiting_user` | planning question, task question 또는 approval을 기다린다. | `open`, `confirmed`, `closed`, `failed` |
| `confirmed` | exact draft가 active intent/queue로 materialize되었다. | `open`, `closed`, `failed` |
| `closed` | 더 이상 mutation action을 받지 않는다. | 없음 |
| `failed` | canonical inconsistency로 mutation이 격리되었다. | recovery가 만든 `open` 또는 `closed`만 |

`closed`와 `failed`에서 들어온 mutation action은 `action.rejected`로 기록하고
state를 바꾸지 않아야 한다.

### 4.4 Draft state

Draft는 mutable document가 아니라 immutable revision chain이다. 화면의
"현재 draft"는 가장 최근 accepted revision을 가리키는 projection이다.
`proposed`는 검증 전 proposal 상태이며 head가 아니다. 최초 proposal이
검증을 통과하면 core는 같은 transition에서 `draft.proposed` provenance event와
`accepted` revision을 함께 기록해 그 revision을 첫 head로 만든다. 이후 revision이
head가 되면 이전 accepted revision의 저장 상태는 `superseded`다.

| state | 의미 | runnable |
|---|---|---|
| `proposed` | worker 또는 user가 patch를 제안했다. | never |
| `accepted` | core가 patch를 검증하고 새 immutable revision을 기록했다. | never |
| `rejected` | 제안이 거절되어 head를 바꾸지 않았다. | never |
| `superseded` | 이후 accepted revision이 head가 되었다. | never |
| `confirming` | confirm prepare가 기록되었지만 commit 전이다. | never |
| `confirmed` | activation receipt가 commit되었다. | confirmed queue gate를 통과한 task만 |

Draft content는 최소한 summary, allowed scope, out-of-scope, acceptance,
ambiguity, task definitions, dependencies, routing intent와 validation intent를
포함한다. UI-only selection, scroll position, open panel은 draft content가 아니다.

### 4.5 Task와 attempt lifecycle

Task lifecycle은 기존 `queued`, `running`, `done`, `blocked`, `failed`,
`needs_user`, `partial`, `deferred`를 유지한다. 현재 의미와 scheduler 분류는
`src/schemas.rs:230-308`, `src/schemas.rs:622-660`이 근거다.

Attempt lifecycle은 task lifecycle과 분리한다.

| attempt state | 의미 | 다음 상태 |
|---|---|---|
| `prepared` | id와 packet/evidence anchor가 durable하지만 spawn 전이다. | `running`, `cancelled`, `abandoned` |
| `running` | worker invocation이 시작되었다. | `needs_user`, `succeeded`, `failed`, `timed_out`, `cancelled`, `abandoned` |
| `needs_user` | 이 attempt가 question을 남기고 멈췄다. | terminal, answer는 새 continuation attempt를 causally 연다. |
| `succeeded` | result가 존재하고 evaluation/finalization 입력이 되었다. | terminal |
| `failed` | invocation 또는 result/evaluation이 실패했다. | terminal |
| `timed_out` | wall limit으로 종료되었다. | terminal |
| `cancelled` | user stop이 확인되었다. | terminal |
| `abandoned` | process가 사라졌고 usable result도 없다. | terminal |

한 attempt를 다시 `running`으로 되돌리면 안 된다. provider native resume도
동일한 invocation을 기술적으로 이어 붙일 뿐, Yardlet retry/answer/redirect가
새 core action을 수반하면 새 `attempt_id`를 MUST 만든다. 동일한
`worker_session_ref`를 재사용할 수는 있다.

## 5. Event 계약

### 5.1 Event envelope

모든 normalized event는 다음 필드를 MUST 가진다.

```yaml
schema_version: 1
event_id: evt_...
session_id: ses_...
seq: 17
type: question.asked
recorded_at: "2026-07-12T14:00:00Z"
actor:
  kind: worker
  id: codex
action_id: act_...
causation_id: evt_...
correlation_id: cor_...
task_id: task_...
attempt_id: att_...
payload: {}
raw_ref:
  artifact_id: art_...
  byte_start: 120
  byte_end: 248
```

- `event_id`, `session_id`, `seq`, `type`, `recorded_at`, `actor`, `payload`는
  MUST다.
- `recorded_at`은 감사 표시용이며 ordering에 사용하면 안 된다.
- task 관련 event는 `task_id`를, worker invocation 관련 event는
  `attempt_id`를 MUST 가진다.
- action으로 생긴 event는 `action_id`를 MUST 가진다.
- 다른 사실의 결과인 event는 `causation_id`를 MUST 가진다.
- normalized worker event는 가능한 경우 `raw_ref`를 MUST 가진다. text-only
  worker는 한 raw span을 `worker.message`로 normalize해도 된다.
- private chain-of-thought를 만들거나 추론해 저장하면 안 된다. provider가
  공개한 message/progress/tool event와 raw output만 기록한다.

### 5.2 Stable event vocabulary

| event type | 필수 linkage | 의미 |
|---|---|---|
| `session.opened`, `session.closed` | session | durable scope 생성/종료 |
| `draft.proposed` | session, revision | 검증 전 proposal provenance. 유효하면 같은 transition에서 해당 revision이 `accepted` head가 된다. |
| `draft.revised` | session, parent/new revision, action | validated head 변경 |
| `draft.rejected` | proposal, action | head 불변 |
| `draft.confirm.prepared` | revision, confirm action | materialization 준비, non-runnable |
| `draft.confirmed` | revision, confirmation, action | activation commit |
| `task.materialized` | task, confirmation | confirmed draft에서 task 생성 |
| `task.state.changed` | task, action 또는 attempt | 기존 typed task transition |
| `attempt.prepared` | task, attempt | invocation 전 durable record |
| `worker.started` | task, attempt | worker process 시작 |
| `worker.message` | task, attempt, raw ref | user-visible worker text/progress |
| `tool.started`, `tool.completed` | task, attempt, raw ref | provider가 노출한 tool activity |
| `worker.checkpoint` | task, attempt, artifact | continuation anchor |
| `question.asked` | task, attempt, question | exact open question 생성 |
| `user.answered` | task, question, answer, action | answer 기록 |
| `artifact.created` | task, attempt, artifact | durable output 등록 |
| `resource.declared` | task, attempt, resource | worker proposal를 core가 기록 |
| `resource.observed` | resource, observation | probe 결과와 `observed_at` 기록 |
| `resource.state.changed` | resource, action/observation | lifecycle 변경 |
| `validation.started`, `validation.completed` | task, attempt | deterministic validation 결과 |
| `worker.completed` | task, attempt | invocation terminal outcome. `payload.result`는 section 4.5 attempt terminal state 어휘를 MUST 쓴다. |
| `completion.recorded` | task, attempt, completion | evaluated task outcome |
| `action.requested`, `action.completed`, `action.rejected` | action | surface-neutral operation receipt |

새 event type은 additive minor evolution으로 추가할 수 있다. 모르는 event가
없어도 안전하게 계산되는 projection만 계속할 수 있으며, runnability나
ownership에 필요한 unknown event는 section 12의 fail-closed 규칙을 따른다.

## 6. Action 계약

Surface는 파일을 직접 수정하지 않고 다음 core action을 dispatch한다. 모든
mutation action은 caller가 생성한 stable `action_id`, 대상 id, expected
revision 또는 expected state, payload를 MUST 포함한다.

| action | precondition | canonical effect | invalid 처리 |
|---|---|---|---|
| `revise` | open planning session, expected head 일치 | validated draft revision 추가 | stale head면 reject, 자동 rebase 금지 |
| `confirm` | accepted draft head, open session, user authority | prepare, intent/queue materialize, activation commit | mismatch/partial이면 non-runnable reject 또는 recovery |
| `answer` | open `question_id`, unanswered | answer event와 continuation causality 기록 | closed/duplicate answer면 prior result 반환 또는 conflict |
| `approve` | open approval gate | scoped grant receipt | 다른 task/gate로 재사용 금지 |
| `redirect` | running/needs_user task | stop/checkpoint reason, 새 attempt guidance | subprocess가 live chat을 받았다고 가장 금지 |
| `pause` | running session/task drain | scheduler hold 기록 | worker process pause 지원과 혼동 금지 |
| `resume` | paused drain 또는 supported worker session | hold 해제 또는 새 continuation attempt | stale worker session이면 explicit fallback |
| `retry` | terminal retryable attempt/task | 새 attempt 생성 | 기존 attempt id 재사용 금지 |
| `stop` | owned live attempt/resource | stop requested와 observed terminal state 기록 | PID 존재만으로 external process kill 금지 |
| `open` | artifact/resource 존재 | viewer/attach action result | live 여부 미확인이면 last-observed 표시 |
| `cleanup` | Yardlet ownership과 lifecycle 확인 | owned resource 정리 결과 | external/unknown ownership이면 reject |

`defer`, `revive`, task add, follow-up ingest처럼 현재 존재하는 operation도
같은 action envelope를 SHOULD 사용한다. 단, active queue에 대한 명시적
user task add는 planning draft promotion이 아니다. planning에서 유래한 새
task를 active queue로 넣는 경로는 반드시 새 draft revision과 `confirm`을
거쳐야 한다.

`yardlet goal` express path는 예외적인 auto-confirm이 아니다. 이미 bounded한
goal을 제출하는 user operation 자체가 exact generated draft를 즉시 confirm하는
명시적 `confirm` action으로 모델링되어야 한다. core는 draft revision과 confirm
receipt를 모두 기록해야 한다.

## 7. Surface projection 계약

| consumer | 읽는 projection | dispatch 가능한 action | 금지 사항 |
|---|---|---|---|
| CLI | session summary, current draft diff, queue/task state, channel, attempt/resource list | 공통 core action 전체 중 CLI가 노출한 subset | 파일 직접 수정, CLI-only state |
| TUI | CLI와 같은 state에 selection/scroll을 더한 ephemeral view | 같은 action endpoint | key handler 안의 별도 transition |
| GUI | 같은 state에 rich preview를 더한 view | 같은 action endpoint | GUI DB를 canonical state로 사용 |
| conversational planning | planning channel, draft revision chain, semantic diff | `revise`, `answer`, `confirm` | low ambiguity를 이유로 silent confirm |
| durable task channel | task events를 seq로 merge하고 attempt boundary를 보존 | `answer`, `redirect`, `retry`, `stop`, `approve` | retry stream 덮어쓰기 |
| runtime resources | task/attempt별 resource와 last observation | `open`, `stop`, `resume`, `cleanup` | probe 없이 live 표시 |
| generic worker adapter | packet, attempt identity, raw event sink, result proposal path | worker event/result 제출만 | canonical id/seq/state 직접 기록 |

Surface-local ephemeral state는 선택, scroll, panel 크기, 임시 filter만 MAY
포함한다. 그것이 workflow 결과나 runnability를 바꾸면 canonical action으로
승격해야 한다.

## 8. Identity, ordering, ownership, provenance, causality

### 8.1 ID 규칙

- 모든 canonical entity id는 workspace 안에서 영구적으로 unique해야 한다.
- id 문자열은 opaque다. timestamp나 prefix parsing으로 ordering, type,
  ownership을 추론하면 안 된다.
- 권장 prefix는 `ses_`, `evt_`, `act_`, `drv_`, `cnf_`, `int_`, `que_`,
  `tsk_`, `att_`, `qst_`, `ans_`, `art_`, `res_`, `cmp_`다.
- 구현은 UUIDv7, ULID 또는 동등한 collision-resistant id를 MAY 사용하지만
  ordering은 반드시 `seq`와 명시적 relation을 사용한다.
- fixture는 고정 id를 사용해야 하며 random generator나 현재 시간에 의존하면
  안 된다.
- legacy `intent_id`, `queue_id`, `task.id`, `run_id`는 rewrite하지 않는다.
  새 projection에서 각각 core id로 취급한다. `attempt_id`는 legacy run의
  `run_id`와 동일한 compatibility alias를 MUST 제공한다.
- provider의 worker session id는 `worker_session_ref`다. `attempt_id`나
  `session_id`로 재사용하면 안 된다.

### 8.2 Ordering 규칙

- core는 session별 positive integer `seq`를 한 번만, strictly increasing하게
  부여해야 한다.
- event replay의 유일한 기본 정렬 키는 `(session_id, seq)`다.
- 한 session에서 동일 `seq`의 서로 다른 event는 invalid다.
- 같은 `event_id`의 byte-equivalent canonical payload는 duplicate로 무시한다.
  같은 id에 다른 payload는 corruption이다.
- 정상 writer가 만든 journal은 contiguous seq를 SHOULD 가진다. gap은 missing
  record로 보고 gap 뒤의 mutation projection을 fail closed한다.
- 서로 다른 session의 event는 `causation_id` 또는 explicit link가 없으면
  상대 순서를 추론하지 않는다.
- queue display priority, filesystem order, RFC3339 time, directory suffix는
  event order를 대체하지 못한다. 현재 code도 latest run 선택 시 directory
  이름만 쓰지 않고 typed start time과 metadata를 함께 본다
  (`src/run.rs:1356-1389`). 새 계약은 이를 더 강하게 seq로 고정한다.

### 8.3 Ownership과 provenance

- `actor`는 `system`, `user`, `worker`, `surface` 중 하나이며 worker/surface는
  stable id를 MUST 포함한다.
- worker-authored payload는 `producer.worker_id`와 `attempt_id`를 MUST 가진다.
- core-derived record는 input event/action ids와 algorithm/schema version을
  SHOULD 기록한다.
- artifact는 content digest, path, media type, role, exact producer attempt를
  MUST 기록한다. path만 같은 새 content는 새 artifact revision 또는 새 id다.
- resource는 `ownership = yardlet | worker | user | external | unknown`을 MUST
  기록한다. `external`과 `unknown`은 cleanup 대상이 아니다.
- completion은 worker self-report만으로 `done`이 될 수 없다. evaluator와
  validation evidence ref를 MUST 가진다. 이는 current finalize/evaluator
  boundary (`src/run.rs:2655-2739`)를 유지한다.

### 8.4 Causality와 idempotency

- `action.requested`가 직접 만든 모든 event는 같은 `action_id`를 가진다.
- `causation_id`는 직접 원인 event 하나를, `correlation_id`는 planning-to-
  completion처럼 전체 흐름 하나를 묶는다.
- answer는 question을, continuation attempt는 answer 또는 retry/redirect
  action을, artifact와 completion은 producer attempt를 causation chain으로
  MUST 가리킨다.
- action handler는 `action_id -> request_digest -> terminal receipt`를
  canonical하게 저장해야 한다. 같은 id와 같은 digest 재요청은 기존 receipt를
  반환하고 새 event를 만들지 않는다.
- 같은 `action_id`와 다른 digest는 `idempotency_conflict`로 reject한다.
- projection은 event id를 dedupe해야 한다. duplicate delivery, replay,
  restart는 entity, turn, artifact, transition을 중복 생성하면 안 된다.
- index rebuild는 canonical record를 수정하거나 새 canonical event를 만들면
  안 된다.

## 9. Draft에서 runnable queue까지의 유일한 경로

### 9.1 전이표

| from | input | guard | writes | to | runnable |
|---|---|---|---|---|---|
| no draft | worker/user proposal | schema/content valid | `draft.proposed` provenance와 같은 revision의 `accepted` snapshot | `accepted` first head | no |
| accepted Rn | `revise(expected=Rn)` | expected head match | Rn+1, `draft.revised` | Rn superseded, Rn+1 accepted | no |
| accepted Rn | reject proposal | proposal exists | `draft.rejected` | Rn unchanged | no |
| accepted Rn | `confirm(expected=Rn)` | session open, exact digest valid, user authority | `action.requested`, `draft.confirm.prepared` | `confirming` | no |
| confirming Rn | materialize | prepare valid | intent and queue snapshots carrying confirmation id/digests | `confirming` | no |
| confirming Rn | commit activation | both snapshots validate against prepare | activation receipt, `draft.confirmed`, `action.completed` | `confirmed` | yes, after scheduler gate |
| confirming Rn | mismatch/corruption | any digest or id mismatch | `action.rejected` or recovery fault | accepted or failed | no |
| confirmed Rn | revise request | active work mutation policy | new draft branch only | accepted Rn+1 | existing confirmed queue only |

### 9.2 Confirm 우회 금지 불변식

다음 조건을 모두 만족할 때만 task를 `Runnable`로 분류할 수 있다.

```text
queue.intent_id == intent.id
queue.confirmation_id == intent.confirmation_id
activation.confirmation_id == queue.confirmation_id
activation.draft_revision_id == confirmed draft head
activation.intent_digest == digest(intent)
activation.queue_digest == digest(queue)
activation.status == committed
task.materialized_by_confirmation_id == activation.confirmation_id
normal task dependency/approval/capability gates pass
```

하나라도 없거나 다르면 `recorded_state: queued`를 표시할 수는 있어도
`runnable: false`, `runnable_reason: unconfirmed_or_inconsistent`여야 한다.

Planner result, draft accepted event, low ambiguity, empty open question,
surface-local confirm flag, queue 파일 존재, `TaskState::Queued` 단독으로는
confirm 증거가 아니다. crash/restart 후에도 같은 규칙을 MUST 재적용한다.

## 10. Persistence boundary

### 10.1 Canonical records

다음은 canonical이며 core만 쓴다.

| record family | 역할 | mutation model |
|---|---|---|
| session manifest | session identity/lifecycle | versioned snapshot + action receipt |
| event journal/segments | ordered facts | append-only, immutable sealed segment |
| action receipts | idempotency와 commit/reject 결과 | one terminal receipt per action id |
| draft revisions | exact planning content | immutable |
| activation receipts | confirm commit point | immutable |
| active intent/queue snapshots | confirmed execution contract | core-written versioned snapshots |
| attempt `run.yaml`, result/evaluation/raw log | attempt accountability와 raw evidence | append artifacts + atomically sealed lifecycle |
| question/answer facts | user decision history | append-only facts |
| artifact/resource declarations와 observations | work-owned output/liveness | immutable declarations + append observations |
| completion record | evaluated terminal outcome | immutable |

현재 `intent-contract.yaml`, `work-queue.yaml`, per-run files, conversations,
transitions는 migration 전까지 canonical legacy records다. V010 implementation은
이들을 조용히 derived cache로 재분류하거나 rewrite하면 안 된다.

### 10.2 Derived index와 cache

다음은 derived이며 언제든 삭제하고 rebuild할 수 있어야 한다.

- session list와 latest-session pointer
- task channel index와 latest event offset
- task-to-attempt, task-to-artifact, task-to-resource reverse index
- resource live-status summary와 preview cache
- search index, unread count, surface navigation state
- telemetry aggregate, skill/routing score
- materialized UI view model

Derived record는 `derived_from` canonical ids와 highest applied seq를 SHOULD
기록한다. index가 없거나 stale하거나 malformed이면 canonical replay로
read-only projection을 만들고, canonical data를 수정하지 않는다. rebuild
실패는 execution state를 바꾸지 않는다.

### 10.3 Durability와 commit point

- canonical snapshot과 receipt는 same-directory temp file, file `sync_all`,
  atomic rename을 MUST 사용한다. 가능하면 parent directory sync도 SHOULD 한다.
- event는 complete record 단위로 append하거나 immutable segment를 atomic rename
  해야 한다. partial tail은 마지막 valid boundary까지만 읽고 mutation을
  fail closed한다.
- confirm처럼 여러 record를 묶는 operation은 prepare receipt와 마지막
  activation commit receipt를 MUST 사용한다. intent/queue 파일 존재는 commit이
  아니다.
- raw log append failure와 canonical event failure를 구분한다. canonical event를
  기록하지 못한 mutation은 성공으로 surface에 반환하면 안 된다.
- single core writer가 workspace mutation을 serialize해야 한다. 병렬 worker는
  각 attempt directory에만 proposal/raw result를 쓰고, queue/session commit은
  orchestrator가 순차 적용한다. 현재 parallel invariant가 같은 방향이다
  (`src/parallel.rs:1-17`, `src/parallel.rs:409-423`).

## 11. Deterministic replay와 restart recovery

### 11.1 Replay algorithm

Replay implementation은 다음 순서를 MUST 따른다.

1. session manifest, activation, active snapshots와 event segment의 supported
   schema를 검증한다.
2. canonical record digest와 referential integrity를 검증한다.
3. event를 `event_id`로 dedupe한다. conflicting duplicate는 격리한다.
4. session별 `seq`로만 정렬한다. gap/conflict 뒤의 mutation fold는 중단한다.
5. `action_id` terminal receipt를 먼저 구성해 incomplete action을 식별한다.
6. draft head와 confirm activation을 fold한다.
7. confirmed task를 materialize하고 task state transition을 fold한다.
8. attempt, raw event refs, question/answer, artifact/resource, completion을
   exact linkage로 fold한다.
9. task별 event를 seq order로 merge하되 attempt boundary를 보존해 정확히
   하나의 `TaskChannel`을 만든다.
10. derived indexes는 결과에서 새로 만들되 canonical input을 쓰지 않는다.

동일 canonical input bytes와 동일 supported schema set은 byte-equivalent
normalized snapshot을 MUST 만든다. display-only timestamp formatting과 localized
label은 snapshot 바깥 surface concern이다.

### 11.2 Crash window와 안전한 recovery

| crash window | restart 판단 | 안전한 기본 동작 |
|---|---|---|
| action request 전 | canonical effect 없음 | no-op |
| `action.requested` 후 mutation 전 | terminal receipt 없음 | action별 recover 또는 reject, duplicate effect 금지 |
| confirm prepare 후 snapshot 전 | activation 없음 | non-runnable, exact revision으로 materialize 재개 가능 |
| intent만 기록, queue 전 | prepare와 digest 비교 | non-runnable, queue 재생성 또는 전체 abort |
| intent/queue 기록, activation 전 | commit receipt 없음 | non-runnable, digest 일치 시 commit 재개, 불일치 시 quarantine |
| activation commit 후 derived index 전 | activation valid | runnable 가능, index rebuild |
| attempt prepared 후 spawn 전 | no `worker.started`, no live observation | attempt abandoned/cancelled 기록 후 새 retry action만 허용 |
| worker started, PID/session live | observed live | adopt/monitor, 새 중복 spawn 금지 |
| worker gone, no result | abandoned | attempt seal, task를 정책에 따라 queued/partial로 전이 |
| result 존재, completion 전 | result digest와 attempt linkage valid | evaluator/finalize idempotently 재개 |
| completion 후 queue/index 전 | completion/action receipt valid | task snapshot projection을 재적용, duplicate completion 금지 |
| canonical complete, index missing/corrupt | canonical valid | read-only replay와 index rebuild |

Resource restart recovery는 declared state와 observed state를 분리한다.
`last_observed: live`는 현재 live 주장이 아니다. PID, port, browser/session 또는
service health를 다시 probe하고 새 `resource.observed`를 기록하기 전에는
`status: unknown`, `last_observed_status`, `last_observed_at`을 보여야 한다.

Recovery 자체도 stable `action_id`를 가진 idempotent core action이어야 한다.
같은 orphan/result를 두 번 finalize하거나 같은 queue transition, answer,
artifact, completion을 두 번 만들면 안 된다.

## 12. Version tolerance와 invalid-state 처리

### 12.1 Version 규칙

- 모든 canonical record는 integer `schema_version`을 MUST 가진다.
- 같은 version에서 optional field 추가와 unknown event type 추가는 additive다.
  reader는 unknown field를 무시할 수 있지만 raw record를 보존해야 한다.
- required meaning을 바꾸는 변경은 새 schema version이 필요하다.
- reader가 아는 older version은 explicit adapter를 거쳐 normalized form으로
  읽을 수 있다. adapter는 source를 rewrite하면 안 된다.
- higher unknown version은 opaque read-only record로 보존한다. runnability,
  confirm, ownership, cleanup을 추론하지 않는다.
- unknown event가 display-only면 raw placeholder로 channel에 남길 수 있다.
  unknown event가 draft head, activation, task state, ownership, causality에
  영향을 줄 가능성이 있으면 그 projection은 `unsupported_semantics`로
  fail closed한다.

Serde의 additive default는 현재 partial/legacy read에 이미 쓰이지만
(`src/schemas.rs:1068-1098`, `src/run.rs:115-158`), default value가 실제 증거를
대체하면 안 된다. 특히 empty confirmation, empty ownership, empty attempt
linkage는 확인된 값이 아니라 unknown이다.

### 12.2 Invalid-state matrix

| invalid state | projection | mutation/runnable 처리 |
|---|---|---|
| duplicate id, same digest | 한 번만 적용 | prior receipt 반환 |
| duplicate id, different digest | conflict 표시 | quarantine, mutation reject |
| missing referenced parent/event/action | orphan 표시 | causal subtree non-runnable |
| sequence gap/conflict | gap 전까지만 표시 | gap 이후 mutation fold 중단 |
| queue/intent/activation digest mismatch | 각 raw record 표시 가능 | 모든 task non-runnable |
| task가 unknown intent/confirmation 참조 | legacy/orphan task 표시 | non-runnable |
| answer가 없거나 closed question 참조 | unattached turn 표시 | continuation 금지 |
| artifact/resource에 attempt provenance 없음 | unowned evidence 표시 | completion/cleanup 근거로 사용 금지 |
| malformed derived index | 무시 | canonical replay, state 변화 없음 |
| malformed canonical required record | partial raw evidence 표시 | affected operation fail closed |
| unknown resource ownership | last-observed 정보 표시 | stop/cleanup reject |

## 13. Legacy no-index fallback projection

새 session/channel/resource index가 없는 기존 workspace는 다음 read-only
fallback으로 해석한다.

1. `intent-contract.yaml`과 `work-queue.yaml`을 source snapshot으로 읽는다.
   둘의 `intent_id`가 일치하지 않으면 별도 orphan projection으로 보인다.
2. task namespace는 `(intent_id, task_id)`다. intent id가 비어 있으면
   `legacy:<queue_id 또는 source digest>` namespace를 만든다. source를 수정하지
   않는다.
3. 알려진 per-run `run.yaml`만 attempt로 읽고 `run_id`를 `attempt_id` alias로
   쓴다. task/intent linkage가 맞지 않으면 unattached attempt로 보인다.
4. run ordering은 valid typed `started_at`, filesystem modified time, stable path
   name의 명시적 fallback tuple을 사용한다. 이 순서는 display fallback일 뿐
   causality나 runnability 증거가 아니다. 현재 latest-run 호환 동작은
   `src/run.rs:1356-1389`에 근거한다.
5. `conversations/<task_id>.yaml` turn은 `run_id`가 있으면 exact attempt에,
   없으면 legacy task channel에 unattached turn으로 넣는다.
6. `transitions/<task_id>.yaml`은 matching `intent_id`만 task에 적용한다. empty
   intent id는 legacy 표시를 붙인다. 현재 scoped lookup은
   `src/state.rs:296-311`에 근거한다.
7. result, evaluation, checkpoint, handoff, raw log처럼 알려진 run artifact는
   해당 attempt 아래에 표시한다. 파일 존재만으로 completion이나 ownership을
   합성하지 않는다.
8. exactly one `TaskChannel`을 `(namespace, task_id)`마다 만들고 모든 attempt를
   보존한다.
9. fallback은 index나 canonical source를 쓰지 않는다.
10. activation receipt가 없는 legacy queue의 `TaskState::Queued`는
    `recorded_state: queued`, `runnable: unknown`으로 projection한다. V010 신규
    scheduler는 explicit adoption/compatibility policy 없이는 이를 runnable로
    승격하면 안 된다.

이 read-only fallback은 기존 production scheduler의 현재 동작을 변경하지
않는다. 신규 scheduler 연결과 legacy adoption UX는 후속 implementation에서
결정할 gap이다. 여기서의 안전 기본값은 data loss와 false-runnable을 모두
피하는 것이다.
