# v0.9 mechanism absorption ledger

This document records the final v0.9 disposition of the mechanism-absorption
roadmap. It separates shipped behavior from explicitly deferred or rejected
work. A deferred item has a measurable restart condition; it is not silently
treated as complete.

## Final disposition

| Item | Disposition | Evidence and boundary |
|---|---|---|
| V09-001 explicit task goal condition | Implemented | Tasks carry a backward-compatible goal contract with a condition, feedback-cycle cap, and policy. |
| V09-002 deterministic failure feedback | Implemented | Failed validation and acceptance evidence is recorded and injected into the next bounded attempt; exhausting the persisted cap stops at NeedsUser. |
| V09-003 read-only memory scout fanout | Implemented | `yardlet memory scout` runs topic scouts against isolated copies and writes unapplied candidates; `yardlet memory apply --run <run-id>` is the explicit core write step. |
| V09-004 stale memory candidates | Implemented | `look_at` landmarks use git changes where available and filesystem evidence for uncommitted and non-git changes. |
| V09-005 bounded local heartbeat | Implemented | `yardlet watch` observes a command or path in the foreground with interval, until condition, run cap, duration cap, cancellation, and a JSON run artifact. |
| V09-006 mechanism fixtures | Implemented | `yardlet eval fixtures` runs ten isolated deterministic fixtures with shared human and JSON verdicts and a non-zero exit on failure. |
| V09-007 comparison documents | Implemented | `docs/comparison/` documents agent CLI boundaries, task versus worker yielding, benchmark posture, and draft launch copy. |
| V09-008 TUI inbox and Answer context | Implemented | Home routes answerable stops through one Answer action; the Answer view scrolls the current intent's worker output and conversation with a fallback when output is unavailable. |
| H4 deterministic observation mining | Implemented, narrow by design | `yardlet harness review` surfaces threshold-based no-result worker and high-retry task-kind observations as suggestions. It does not mutate policy. |

## Deferred ledger

### Background or daemon watch

**Disposition:** Deferred beyond v0.9.

The current command is a bounded foreground observer. Persistent scheduling,
remote control, and cloud execution would introduce a different lifecycle and
failure model.

**Restart condition:** A dogfood run artifact shows that a foreground watch was
terminated with its observation goal unmet specifically because the terminal
session ended.

### Automatic application of scout candidates

**Disposition:** Deferred beyond v0.9.

Scout reports are proposals. Automatic application would weaken the visible
boundary between a generative worker and the core's canonical memory writer.

**Restart condition:** An isolated diff fixture proves that automatic apply
preserves the single-writer rule and cannot overwrite an existing memory
document without an explicit conflict outcome.

### Broader H4 mining themes

**Disposition:** Deferred until evidence exists.

Validation-failure themes, NeedsUser themes, forbidden-path attempts, and merge
conflict themes are not mined in v0.9. Adding schemas before repeated evidence
would make the core predict future telemetry.

**Restart condition:** The transition or telemetry ledger contains at least
three instances of one recurring theme that the existing two miners do not
surface. Add only that evidenced theme.

## Rejected for v0.9

### Model benchmarks, consensus voting, and best-of-N

These are outside the deterministic mechanism evaluator. Reconsider after
v1.0 only if repeated dogfood decisions cannot be informed by mechanism
fixtures and workspace telemetry.

### Worker-grade LSP, debugger, or browser tools in the core

These remain worker capabilities. Reconsider only if a concrete acceptance
check cannot be expressed through the existing packet, process, result-file,
and deterministic-evaluator boundary.

## Guarantee boundary

The implemented items guarantee recorded contracts, bounded transitions,
isolated local fixtures, and durable evidence. They do not guarantee that any
worker will produce correct code, that every task can recover without a human
decision, or that one model performs better than another.
